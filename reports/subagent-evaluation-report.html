01 · Verdict

Useful code. Essential review. Expensive orchestration.

The agent system worked best as a controlled implementation-and-review pipeline. It worked poorly when outputs were not retained, when implementers treated self-review as sufficient, or when a reviewer stopped responding.

“The agents earned their keep by catching each other’s mistakes—not by being right the first time.”

Tasks 1–4 reached independent approval. Task 5 reached 110 passing tests and a clean typecheck, but its replacement reviewer found two accessibility defects that remain open because all sub-agents were stopped. The early Luna discovery agents may have helped planning, but the absence of retained, agent-specific outputs makes their individual value impossible to audit.

02 · Operating model

How the sub-agents were used

The root agent remained the coordinator. Sub-agents never received deletion authority, never owned the final completion claim, and were kept serial because each task depended on the same evolving shell and chat surfaces.

01

Luna discovery

Read-only flow mapping, UI inventory, and UX audit before the implementation plan.

02

Root synthesis

Resolved product choices, wrote bounded briefs, defined tests, file limits, non-goals, and commit subjects.

03

Terra build

One heavy implementation task at a time, with RED/GREEN evidence, self-review, report, and commit.

04

Fresh Terra review

A separate read-only agent checked the exact task range for spec, regressions, accessibility, and React correctness.

05

Remediation loop

Findings went back to the original implementer for regression tests and a focused fix; only then did the next task start.

Root-only responsibilities: worktree isolation, plan approval, agent routing, brief construction, review-package generation, progress tracking, integration decisions, final browser verification, cleanup, and any claim that the redesign is complete.

Luna · discovery3

Low-complexity read-only passes. Their task identities are known; their standalone findings are not retained in the worktree.

Terra · implementation5

One agent per design task. All five delivered commits and passing test evidence.

Terra · review7

Six produced usable verdicts or approvals. One Task 5 reviewer stalled and was replaced.

The redesign advanced serially. Every implementation gate depended on the preceding task, which limited safe parallelism but made fault attribution clearer.

06 · Better operating model

How to make future agents materially better

The answer is not more agents. It is smaller assignments, retained evidence, earlier failure detection, and a real browser gate for visual work.

01

Make discovery durable

Every Luna run must write a small, cited artifact. No artifact means no credit and no dependency on the output.

02

Use one implementer

Keep dependent edits serial. Give each Terra agent exact files, invariants, non-goals, test cases, and a fixed completion contract.

03

Time-box reviewers

Require early acknowledgement and a preliminary finding count. Replace an unresponsive reviewer after one missed checkpoint.

04

Review the system, not only files

Add page-level matrices for focus, live regions, hydration, browser globals, reduced motion, persistence invariants, and mutation ownership.

05

Shrink evidence packages

Send the brief, exact changed files, and test summary. Large generated diffs should be optional, not the reviewer’s only entry point.

06

Require visual proof

For design tasks, inspect real pages at 390, 768, 1280, and 1440px. DOM tests cannot approve composition, rhythm, overflow, or polish.
